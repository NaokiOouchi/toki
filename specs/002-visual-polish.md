# 002 — モックアップ準拠の視覚調整

## Why

Phase 1 MVP（spec 001）の実装が完了した時点で、実機表示とモックアップ画像との視覚的な乖離が見つかった。具体的には：

- **時計の外周がウィンドウ内縁ギリギリ**で描画され、モックアップにある「ウィンドウ内縁との余白」が再現できていない
- **SPEC.md §2 と §1 の記述が矛盾**：§1「時計直径 約240px」、§2「内径 220px、外径 260px」。後者は typo の可能性が高く、実装は §2 の数値で書かれている
- **時間トラックの内側リング輪郭線**が薄すぎて、モックアップに見える「リングがある」感が出ていない
- **外側リング輪郭線**を描画しているが、モックアップでは見えず冗長
- **長い次の予定タイトル**（例：「16:00 ENEOS実装」のような長文）が末尾省略されると情報が落ちる。モックアップでは折り返して 2 行で表示されている

放置すると：
- 自分用ツールだが「妙にギリギリで余白が足りない」「次の予定が読めない」体験で完成度が下がる
- SPEC の内部矛盾は将来の他フィーチャーの実装判断時にもノイズになる

## Goal

Phase 1.5（本 polish 完了時）に達成する状態：

1. ウィンドウ内縁とイベント円弧の最外端の間に、視覚的に明確な余白（≥ 20pt）が存在する
2. 時間トラック（内側リング）の境界線が、薄い背景上でも輪郭として認識できる濃度で描画される
3. 外側リングの輪郭線が描画されない（イベント円弧の外端が時間トラックの外側を示す）
4. SPEC.md の時計直径・内径・外径表記が実装と整合し、矛盾が解消されている
5. 「次の予定」ラインが長いタイトルでも省略されず、2 行まで折り返して全文表示される
6. Domain・Infrastructure・Composition 層は無変更（純粋に UI 層の視覚調整のみ）
7. 既存の Domain テスト 36 ケースが引き続き全 pass する

## Non-goals

本 polish では明示的にやらない：

- **色相・フォントの変更**：色 / 太さ / サイズの再設計は対象外
- **針のスタイル変更**：太さ・色・形状は MVP のまま
- **中央テキストの再レイアウト**：3 行構成・フォントサイズは MVP のまま
- **イベント円弧の角丸化 / ピル状化**：モックアップの近似的な見え方であり再現対象外
- **ウィンドウサイズの変更**：280×320 維持
- **メニューバーアイコンのデザイン変更**：MVP の SF Symbol "clock" のまま
- **次の予定 3 行以上の折り返し**：2 行を上限とし、それ以上は末尾省略
- **ホバー時の中央表示切替**：spec 001 同様 Phase 2

## Acceptance Criteria

### 時計の余白
- The イベント円弧の最外端は、ウィンドウ内縁から **20pt 以上** 離れている
- The 時計盤の中心は、ウィンドウの水平中心と垂直中心（時計エリア内）に一致する

### リング輪郭
- The 内側リングの境界線は、light / dark どちらのテーマでも視認可能な濃度で描画される
- The 外側リングの境界線は描画されない
- While イベント円弧が存在しない時間帯について、内側リング輪郭線が時間トラックの内縁を示す

### SPEC 整合性
- The SPEC.md §1「時計直径」表記と §2「内径・外径」表記が、互いに矛盾せず実装値と一致する
- The 表記訂正は別 commit（`docs(spec): ...`）として実装変更とは分離される

### 次の予定ライン
- While 次の予定タイトルと開始時刻の合計が 1 行に収まる長さのとき、1 行で表示される
- While 次の予定タイトルと開始時刻の合計が 1 行に収まらないとき、2 行に折り返して表示される
- If テキストが 2 行にも収まらないとき、then 末尾を省略記号（…）でトリムする
- The 「次」ラベルは常に左端、時刻＋タイトルは右寄せで表示される

### 既存挙動の維持
- The Domain 層（`TimeOfDay` / `Event` / `EventStatus` / `DayTimeline`）には変更が入らない
- The Infrastructure 層（`EventKitGateway`）には変更が入らない
- The Composition 層（`ClockViewModel`）には変更が入らない
- The 既存の Domain テスト 36 ケースは無変更で全 pass する
- The 時計の角度規約（0:00 真上、時計回り）と針の挙動は変わらない
- The イベント円弧のクリック→純正カレンダー.app 起動は引き続き動作する

## Domain Model

本 polish は UI 層のみの視覚調整であり、Domain Model には変更が入らない。

ただし UI 層の幾何情報を保持する `ClockGeometry`（Value Object、Domain ではなく UI 層）の数値が変更される：

```
ClockGeometry (UI 層 Value Object)
  - center: CGPoint
  - innerRadius: CGFloat  // 変更：110 → 100
  - outerRadius: CGFloat  // 変更：130 → 120
  + static func standard(in size: CGSize) -> ClockGeometry
```

**Invariants（不変条件）**：
- `innerRadius < outerRadius`
- 内径と外径の差（時間トラック幅）は 20pt を維持
- `outerRadius + マージン ≤ canvas の最小辺 / 2` で canvas 内に収まる

**変換責務**：
- なし（純粋な視覚値）

## Open Questions

実装着手前に判断したい論点：

1. **内側リング輪郭線の濃度**：α 値・lineWidth をどこに置くか？ 候補は α 0.5〜0.7 / lineWidth 0.5〜1.0pt。実機で見て決めるか、初手で「α 0.6 / lineWidth 0.75pt」と置いて後調整するか
2. **時刻マーク（0/6/12/18）の位置**：現状 `innerRadius - 12 = 88pt` の位置だが、innerRadius が 100 になると `100 - 12 = 88pt`（変わらず）。中央テキストエリアとの距離感を再評価する必要があるか
3. **次の予定ラインの高さ枠**：現状 `.frame(height: 40)` だが、2 行 wrap 時に十分か？ 11pt font × 2 行 + padding なら 36〜40pt 程度で足りる見込みだが、検証が必要
4. **外側ぎりぎりに描画していた時刻ラベルの再評価**：spec 001 では内側配置に修正済み、本 polish で更に動かす必要はあるか
5. **「20pt 以上のマージン」の根拠**：280pt canvas で outer=120 なら端から 20pt 確保できるが、SPEC §1 の「直径 約240px」を尊重するならこのまま、もう少し小さくしても良い
